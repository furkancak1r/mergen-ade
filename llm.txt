Mergen ADE: LLM quick context

Purpose
- Native Rust desktop ADE for managing multiple project terminals.
- Windows-first product. Official releases publish a portable Windows EXE and a signed/notarized macOS ARM64 DMG.
- Not an IDE: no built-in editor, LSP, debugger UI, telemetry, accounts, or cloud state.

Primary entry points
- src/main.rs: application startup and native window bootstrapping.
- src/app.rs: main UI shell and application state. This is the central integration point.
- src/terminal.rs: PTY runtime, terminal emulation snapshots, IO threads, resize/shutdown behavior.
- src/layout.rs: tile grid calculation logic with unit tests.
- src/title.rs: terminal title sanitization and truncation with unit tests.
- src/config.rs: load/save config.toml through ProjectDirs, including legacy compatibility.
- src/models.rs: persisted config and UI enums/types.
- .github/workflows/release.yml: tagged release pipeline for Windows and macOS.

Architecture summary
- UI is built with eframe/egui.
- The main shell combines:
  - activity rail
  - collapsible side panels
  - project explorer / source control / terminal manager views
  - tiled terminal panes in the main area
- PTY sessions are created with portable-pty.
- Terminal parsing/render data comes from tattoy-wezterm-term and tattoy-wezterm-surface.
- Terminal IO and child-process waiting run off the UI thread; app.rs consumes forwarded events and renders cached snapshots.
- Project-level background workers exist for source control refresh and directory indexing. Avoid reintroducing ad hoc UI-thread worker spawning.

Important state boundaries
- src/app.rs owns most user-facing state and orchestration:
  - project list and selected project
  - terminal registry and active terminal
  - source control snapshots/refresh state
  - directory index snapshots
  - persisted UI preferences
- src/terminal.rs should stay focused on terminal runtime and render snapshot production, not broader app policy.
- src/config.rs and src/models.rs define persisted behavior. Schema changes must preserve backward compatibility when possible.

High-risk hotspots
- src/app.rs is large and behavior-dense. Changes here can easily affect:
  - focus/input routing
  - selection/copy behavior
  - pane layout and visibility filtering
  - source control refresh scheduling
  - directory index refresh behavior
- src/terminal.rs is sensitive around:
  - Windows process shutdown and job object cleanup
  - resize timing
  - scrollback/snapshot generation
  - selection/cursor row alignment
- Release/build logic is Windows-oriented by default. Do not casually change target assumptions without checking README and release workflow.

Known behavioral expectations
- Keep UI lightweight and icon-first rather than heavy, button-dense chrome.
- Preserve low-memory, native-first design; avoid heavy new dependencies.
- Keep KNOWN_ISSUES.md append-only unless explicitly asked to rewrite history.
- Reuse shared source-control snapshots across surfaces instead of duplicating git refresh logic.
- Treat config under ProjectDirs as runtime user data, not repository state.

Build and test commands
- cargo build --release
- cargo run --release
- cargo test
- cargo fmt
- Windows portable release build:
  - powershell -ExecutionPolicy Bypass -File .\scripts\build-release.ps1
- If cargo is not on PATH in PowerShell:
  - $env:USERPROFILE\.cargo\bin\cargo.exe <command>

Platform notes
- Repository default local build flow is Windows-oriented.
- Supported release binary path on Windows:
  - target/x86_64-pc-windows-msvc/release/mergen-ade.exe
- Experimental/native macOS build:
  - cargo build --release --target aarch64-apple-darwin
- Windows shell support in config/runtime:
  - PowerShell
  - CMD
- Non-Windows shell support:
  - zsh

Dependencies that matter
- eframe/egui: desktop UI
- portable-pty: terminal sessions
- tattoy-wezterm-term and tattoy-wezterm-surface: terminal model/render support
- crossbeam-channel: event pipelines
- directories + toml + serde: persisted config
- windows-sys: Windows-native shutdown/window behavior

Testing expectations
- Prefer inline unit tests in the same module.
- Test behavior rather than implementation details.
- Minimum bar for feature changes:
  - update or add relevant tests
  - ensure cargo test passes locally
- Existing focused unit-test modules include layout, title, config/model behavior, and some terminal helpers.

Existing docs and their roles
- README.md: product overview, build/release usage, configuration, troubleshooting.
- AGENTS.md: repository workflow instructions for coding agents; treat it as the main agent-policy file.
- KNOWN_ISSUES.md: chronological bug log and regression memory.

Recommended editing strategy for LLM agents
1. Read AGENTS.md first for workflow constraints.
2. Read this file for fast architectural orientation.
3. Open only the directly relevant module(s), usually app.rs plus one focused logic module.
4. Check KNOWN_ISSUES.md before changing terminal selection, source control refresh, layout, or release behavior.
5. Keep changes localized; add tests near the changed logic.

If you want a more standard ecosystem file than llm.txt
- Keep AGENTS.md for agent-specific rules.
- Keep README.md for humans.
- Consider adding llms.txt later as a public, standardized discovery file for external tools. This repo's llm.txt is currently a practical local summary, not a formal web standard.
